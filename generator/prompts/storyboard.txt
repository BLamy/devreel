You are authoring a devreel LESSON: a short "simulated video" that teaches how to
use a library by puppeting a real in-browser dev environment (almostnode) on a
voiceover-synced timeline. You write ONE JavaScript module, `<slug>.lesson.mjs`,
that does `export const lesson = { ... }`. Plain JS (use backtick template
strings for code files). No types, no imports.

GROUND EVERYTHING IN REAL CODE. Use the repo/library digest you were given: real
package names, real exported functions, real APIs, real import paths. Never invent
an API, option, or file that isn't in the digest or the library's real surface. If
unsure, prefer the smallest real example you can verify from the digest.

ONE TOPIC PER LESSON. Each lesson teaches exactly ONE concept. Do NOT bundle topics
(e.g. don't teach useEffect inside a useState lesson, or routing inside a state
lesson). If teaching something well needs more than one lesson, make a SERIES:
give each lesson the SAME `series` title and an ascending `seriesOrder` (1, 2, 3…).
The feed groups a series into one card and plays the parts back-to-back. Prefer a
focused 1–3 minute lesson over a sprawling one.

BUILDING A LIBRARY ACROSS A SERIES. To have later parts start with the code earlier
parts wrote, set `editorSeed` (a map of file → existing content) on the part and only
`type` the NEW code. `workspace.files` should hold the full state for that part (what
the preview/terminal runs). Multiple files are supported — each shows as an editor
tab. When the part RUNS code with `node`, relative imports MUST use explicit
extensions (`import { x } from './src/foo.ts'`) — almostnode's node runtime does not
infer them. Class methods can't be appended across parts (the brace closes), so build
one class in contiguous parts; top-level functions and separate files append cleanly.

THE LESSON SHAPE
  {
    slug, title, subtitle, library, throughline, persona, accent,
    workspace: { files: { "/index.html": "...", "/src/App.jsx": "...", ... },
                 install?: ["pkg", ...], previewPort?: 3000, previewMode?: "react",
                 dbSchema?: "CREATE TABLE ..." },
    diagram?: { nodes: [{id,label,kind}], edges: [{id,from,to,label}] },
    scenes: [ Scene, ... ]
  }

- workspace.files is the FINAL, runnable project (what the preview executes). React
  + react-dom need no install (esbuild transform + import map). Any OTHER npm
  dependency the code imports MUST be listed in workspace.install.
- index.html should load /src/main.jsx which renders the app.
- STYLE WITH TAILWIND: the preview auto-includes the Tailwind Play CDN, so style
  apps with Tailwind utility classes (className="...") — they render with no setup.
  Prefer Tailwind over inline styles. Keep className lines reasonable; the editor
  wraps long lines.

A Scene:
  { id, chapter?, title?, narration, say?, cue?, focus, action? }
- focus is the tool in the spotlight: "editor" | "preview" | "terminal" | "database" | "diagram" | "viz".
- narration: the on-screen caption. Supports **bold**, *italic*, `code`.
- say: OPTIONAL clean spoken line (overrides narration for TTS). Use it whenever the
  narration contains code/symbols so the voice reads naturally (no slashes, braces,
  camelCase spelled out). Keep it conversational, one or two sentences.
- cue: OPTIONAL — a short verbatim phrase from `say` to anchor this scene to the
  audio, or an absolute number of seconds. If omitted, scenes are spaced evenly.
- action.tool MUST equal focus. Actions:
  editor:   { tool:"editor", file:"/src/App.jsx", type:"...new text to TYPE...",
              replace:"...whole new file...", typed?:true, reveal?:[startLine,endLine],
              callouts?:[{line, text, kind?:"info"|"tip"|"warn"}],
              diagnostics?:[{line, column?, endColumn?, message, severity?:"error"|"warning"|"info"}] }
            • Editor files start EMPTY and are built up by each scene's `type`. The
              concatenation of all `type`/`replace` for a file MUST equal
              workspace.files[file] by the end.
            • Show a REAL common mistake with `replace` (the buggy file) + a
              `diagnostics` marker whose `line` matches the buggy line, then a later
              scene `replace`s the corrected file. Diagnostic messages should be the
              actual error a dev would see.
  preview:  { tool:"preview", navigate?:"/", click?:"<css selector>", openEruda?:"network"|"console",
              focusNetwork?:"<url substring>" }
  terminal: { tool:"terminal", run:"npm test", expect?:"<substring of output>" }
  database: { tool:"database", query:"SELECT ...", highlight?:"<column>" }   // needs workspace.dbSchema
  diagram:  { tool:"diagram", reveal?:[ids], highlight?:[ids], activeEdges?:[ids],
              messages?:[{from,to,label}] }                                  // needs lesson.diagram
  viz:      { tool:"viz", animation:"<id>", act?:"<act name>" }

THE VIZ TOOL — 3b1b-style concept animations. Use it to explain the CONCEPT behind
the code: what attention IS before typing an attention block, why momentum escapes
local minima before tuning an optimizer, what convolution means before writing a
filter. Use EXACT ids and act names from the catalog below (source of truth:
`src/viz/manifest.mjs`; the validator rejects unknown animation ids and drops
unknown act names with a warning — the scene then sweeps the whole animation).
- One scene plays ONE act, paced to that scene's narration. Play acts IN ORDER
  across consecutive scenes (each animation is a story: its acts build on each
  other). You may stop early — you don't have to play every act.
- Write the scene's `say` yourself, grounded in what that act shows (the manifest
  summary tells you); do NOT just restate the act name. The animation's visuals
  carry captions already, so the voice should add insight, not read labels.
- A viz beat of 2–5 scenes at the START of a lesson (concept first, then code) or
  right BEFORE the tricky code it explains works best. Don't scatter single viz
  scenes randomly between typing scenes.
- Viz scenes need no workspace files. A lesson may even be pure viz (a concept
  explainer with no code) — then workspace.files can be empty.

SCENE DISCIPLINE — be thorough, never rushed. The viewer must understand EVERY line.
- TYPE IN SMALL PIECES: one scene per line or tiny cluster (an import, one const, one
  chained call, one JSX element). NEVER dump a big block in a single scene.
- EXPLAIN EVERY LINE in that scene's `say` — what it does and WHY. For any API/function,
  say what it does: e.g. what `pipe` does (feeds a value left-to-right through
  functions), what `Effect.map` does, what `Effect.runSync` does, what a `useEffect`
  dependency array means. For TAILWIND classes, name the key classes and what each does
  and why you chose it (e.g. "p-8 adds padding so it isn't cramped; rounded-lg gives
  rounded corners; bg-indigo-600 is the button color").
- SYNC: a scene's narration must describe exactly the code typed in that scene, so the
  voice lines up with the writing. One scene = one small piece + its explanation.
- Code + output appear TOGETHER (editor beside the preview/terminal/db on desktop,
  stacked on mobile). Set `stage` frame sizes to fit the demo — e.g. a one-command
  terminal can be short: stage: { editor: 0.62, mobileEditor: 0.72 }.
- Include one "common bug → fix" beat (editor `diagnostics`), and END by RUNNING it:
  a preview interaction (cursor click / open devtools network + click the request), a
  terminal command, or a DB query.
- `say` is SPOKEN — natural full sentences, never read symbols literally (expand them:
  "useState of zero", "dot then", "arrow function"). `narration` is the on-screen
  caption — short. Put a verbatim phrase from `say` in `cue` so audio aligns.
- Prefer completeness over brevity: many short, well-explained scenes beats a few dense
  ones. A 10–20 scene lesson is normal.
- To build a file incrementally with consistent line numbers, define the final file as
  ordered segments, type one segment per scene, and compute callout/diagnostic line
  numbers from the final text (e.g. find the line containing "onClick"). See the
  existing src/lessons/*.lesson.mjs for the pattern.
